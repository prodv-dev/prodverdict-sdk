The ProdVerdict vision revolves around building a **production contract system** for AIвЂ‘assisted development.  The effort is sequenced into clear phases, each delivering a standalone contract with immediate value while laying the groundwork for a unified platform.

### PhaseВ 1 вЂ“ Access Contract (PaidвЂ‘Access Verification)

**Goal:** catch revenue leakage and customer lockвЂ‘outs by verifying that subscription state in Stripe (or the source of truth) matches the applicationвЂ™s access controls.  This first contract proves the concept and monetizes early because it is tied directly to money.

* **Scope:** compare Stripe customer/subscription state against a readвЂ‘only copy of the app database.  Identify users with active subscriptions who are marked free in the app, and users flagged as paid while their Stripe subscription is cancelled or unpaid.  Detect duplicate customers and mismatched priceвЂ‘toвЂ‘plan mappings.  Implement a CLI and GitHub Action that runs on pull requests and on a schedule.
* **Output:** a deterministic report summarizing violations and providing agentвЂ‘readable instructions (e.g. вЂњmodify the `cancel` handler to revoke accessвЂќ).  The contract fails the CI check if highвЂ‘severity mismatches are found.

### PhaseВ 2 вЂ“ Config Contract (Environment Drift)

**Goal:** prevent deployment failures caused by missing or misconfigured environment variables.  This contract ensures that every variable referenced in code exists across local `.env` files, CI secrets, preview deployments and production.

* **Scope:** scan the repository for references to `process.env.*`, `import.meta.env` and equivalent patterns.  Compare the discovered names against `.env.example` and the providerвЂ™s environment (Vercel, Netlify, Railway or GitHub Secrets).  Detect variables that are declared but never used, and variables used only locally but not defined in production.  Provide suggestions (e.g. вЂњadd `STRIPE_PORTAL_RETURN_URL` to GitHub SecretsвЂќ).
* **Output:** CI comments listing missing or inconsistent variables.  For agent workflows, provide a structured JSON result with fields listing missing variables, the environments affected and steps to fix.
* **Rationale:** agentвЂ‘generated code often introduces new configuration variables without updating deployment secrets.  A deterministic contract prevents вЂњit works on my laptopвЂќ scenarios and complements the paidвЂ‘access contract.

### PhaseВ 3 вЂ“ Migration Contract (Database Safety)

**Goal:** stop dangerous database migrationsвЂ”especially Postgres DDLвЂ”before they reach production.  Many migrations generated by agents or ORMs lock tables, rewrite data or break constraints when applied to large live tables.

* **Scope:** analyse migration files from tools like Prisma, Drizzle or raw SQL.  Use documented Postgres lock semantics to detect statements that acquire `ACCESSВ EXCLUSIVE` locks (`ALTERВ TABLE вЂ¦ ADDВ COLUMN NOTВ NULL`, `CREATEВ INDEX` without `CONCURRENTLY`, etc.).  When live metadata is available (e.g. connecting readвЂ‘only to production), estimate table size to compute blast radius.  Suggest safe rewrites such as using `CREATEВ INDEXВ CONCURRENTLY` or splitting schema changes into multiple migrations.
* **Output:** risk ratings (high, medium, low) with reasons and safe rewrite snippets.  If the risk is high, the contract fails the CI check and provides instructions for the agent/human to rewrite the migration.  The longвЂ‘term plan includes ORMвЂ‘specific codemods that automatically apply safe rewrites.

Postgres migrations are an area where a single wrong statement can lock a production table for minutes or hours.  Combining static linting with live metadata provides a safety net that general linters lack.  While the market has competitors, a narrower, contractвЂ‘based approach emphasises enforcement rather than advisory.

### PhaseВ 4 вЂ“ Boundary Contract (API and MassвЂ‘Assignment Safety)

**Goal:** detect massвЂ‘assignment vulnerabilities and overвЂ‘exposure of sensitive fields.  AIвЂ‘generated route handlers may accept arbitrary JSON fields or return objects with confidential properties.

* **Scope:** parse OpenAPI/GraphQL schemas or infer endpoints from code.  Define lists of forbidden writable fields (e.g. `is_admin`, `plan`, `stripe_customer_id`) and forbidden response fields (e.g. `password_hash`, `reset_token`).  In CI, send test requests with extra fields and compare responses.  If the API persists or returns forbidden fields, the contract fails.
* **Output:** reports showing the offending endpoint and field, with suggestions to whitelist fields or drop sensitive properties from the response.  Provide JSON output for agents and PR comments for humans.
* **Rationale:** OWASP identifies broken object propertyвЂ‘level authorization as a top API risk.  AI agents commonly implement handlers by blindly updating `req.body`, leading to silent privilege escalation.  A deterministic contract stops such risks early.

### PhaseВ 5 вЂ“ Webhook & Restore Contracts (Later Phases)

Subsequent phases will introduce additional contracts:

* **Webhook Contract:** record, replay and validate thirdвЂ‘party webhooks (Stripe, GitHub, Shopify).  Enforce idempotency, signature verification and ordering.  Good when AI agents generate or modify webhook handlers.
* **Restore Contract:** verify that backups actually restore and pass smoke tests.  Run `pg_dump`/`pg_restore` inside CI, execute smoke SQL checks (e.g. ensure `users` and `subscriptions` tables are populated), and record the result.  Only after the restore contract passes does the app get deployed.
* **Configuration Extensions:** integrate additional providers (Supabase, Neon, WorkOS entitlements) and support nonвЂ‘SQL databases.

Each phase is designed to deliver value on its own and to integrate into the unified ProdVerdict engine, with consistent interfaces (CLI, GitHub Action, MCP) and shared policies.

### v0.11.0 вЂ” Stripe Entitlements migration wedge (2026-06)

**Goal:** convert the biggest existential threat (Stripe Entitlements absorbing the "DB disagrees with Stripe" category) into the defensible moat. Be the tool that verifies migrations TO Stripe Entitlements.

* **`entitlements-migration` contract** вЂ” catches users paid in DB but not granted in Stripe (high), stale grants (medium), duplicate grants per customer (medium), and users missing `stripe_customer_id` (high, blocks migration).
* **Access contract with `source_of_truth: stripe_entitlements`** вЂ” verifies the steady state after migration. Compares active Entitlements grants against the DB.
* **`prodverdict scheduled` subcommand** вЂ” prints the recommended GitHub Actions workflow for scheduled drift detection. Access is a scheduled check, not a PR gate.
* **Scheduled-first positioning** вЂ” README quickstart leads with scheduled drift detection, not PR gates. See `scheduled-vs-pr.md`.
* **Pricing reconciliation** вЂ” docs now match the live 2-tier site (Free + Pro Cloud $39/project/mo).
* **AI-trust framing removed** вЂ” ProdVerdict verifies production state, not AI-generated code. The pitch leads with billing drift.
