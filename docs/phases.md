## Product Phases

The ProdVerdict vision revolves around building a **production contract system** for AI‑assisted development.  The effort is sequenced into clear phases, each delivering a standalone contract with immediate value while laying the groundwork for a unified platform.

### Phase 1 – Access Contract (Paid‑Access Verification)

**Goal:** catch revenue leakage and customer lock‑outs by verifying that subscription state in Stripe (or the source of truth) matches the application’s access controls.  This first contract proves the concept and monetizes early because it is tied directly to money.

* **Scope:** compare Stripe customer/subscription state against a read‑only copy of the app database.  Identify users with active subscriptions who are marked free in the app, and users flagged as paid while their Stripe subscription is cancelled or unpaid.  Detect duplicate customers and mismatched price‑to‑plan mappings.  Implement a CLI and GitHub Action that runs on pull requests and on a schedule.
* **Output:** a deterministic report summarizing violations and providing agent‑readable instructions (e.g. “modify the `cancel` handler to revoke access”).  The contract fails the CI check if high‑severity mismatches are found.
* **Rationale:** AI adoption is nearly universal—85 % of developers regularly use AI coding tools【151237730755593†L107-L115】—and 20 million developers use GitHub Copilot【889058447933580†L151-L155】.  When teams rely on agents to write billing logic, accidental access drift becomes common.  A paid‑access contract directly ties to revenue and shows immediate value.

### Phase 2 – Config Contract (Environment Drift)

**Goal:** prevent deployment failures caused by missing or misconfigured environment variables.  This contract ensures that every variable referenced in code exists across local `.env` files, CI secrets, preview deployments and production.

* **Scope:** scan the repository for references to `process.env.*`, `import.meta.env` and equivalent patterns.  Compare the discovered names against `.env.example` and the provider’s environment (Vercel, Netlify, Railway or GitHub Secrets).  Detect variables that are declared but never used, and variables used only locally but not defined in production.  Provide suggestions (e.g. “add `STRIPE_PORTAL_RETURN_URL` to GitHub Secrets”).
* **Output:** CI comments listing missing or inconsistent variables.  For agent workflows, provide a structured JSON result with fields listing missing variables, the environments affected and steps to fix.
* **Rationale:** agent‑generated code often introduces new configuration variables without updating deployment secrets.  A deterministic contract prevents “it works on my laptop” scenarios and complements the paid‑access contract.

### Phase 3 – Migration Contract (Database Safety)

**Goal:** stop dangerous database migrations—especially Postgres DDL—before they reach production.  Many migrations generated by agents or ORMs lock tables, rewrite data or break constraints when applied to large live tables.

* **Scope:** analyse migration files from tools like Prisma, Drizzle or raw SQL.  Use documented Postgres lock semantics to detect statements that acquire `ACCESS EXCLUSIVE` locks (`ALTER TABLE … ADD COLUMN NOT NULL`, `CREATE INDEX` without `CONCURRENTLY`, etc.).  When live metadata is available (e.g. connecting read‑only to production), estimate table size to compute blast radius.  Suggest safe rewrites such as using `CREATE INDEX CONCURRENTLY` or splitting schema changes into multiple migrations.
* **Output:** risk ratings (high, medium, low) with reasons and safe rewrite snippets.  If the risk is high, the contract fails the CI check and provides instructions for the agent/human to rewrite the migration.  The long‑term plan includes ORM‑specific codemods that automatically apply safe rewrites.
* **Rationale:** security research shows that AI‑generated code introduces vulnerabilities at much higher rates【50550606903651†L20-L27】.  Postgres migrations are an area where a single wrong statement can lock a production table for minutes or hours.  Combining static linting with live metadata provides a safety net that general linters lack.  While the market has competitors, a narrower, contract‑based approach emphasises enforcement rather than advisory.

### Phase 4 – Boundary Contract (API and Mass‑Assignment Safety)

**Goal:** detect mass‑assignment vulnerabilities and over‑exposure of sensitive fields.  AI‑generated route handlers may accept arbitrary JSON fields or return objects with confidential properties.

* **Scope:** parse OpenAPI/GraphQL schemas or infer endpoints from code.  Define lists of forbidden writable fields (e.g. `is_admin`, `plan`, `stripe_customer_id`) and forbidden response fields (e.g. `password_hash`, `reset_token`).  In CI, send test requests with extra fields and compare responses.  If the API persists or returns forbidden fields, the contract fails.
* **Output:** reports showing the offending endpoint and field, with suggestions to whitelist fields or drop sensitive properties from the response.  Provide JSON output for agents and PR comments for humans.
* **Rationale:** OWASP identifies broken object property‑level authorization as a top API risk.  AI agents commonly implement handlers by blindly updating `req.body`, leading to silent privilege escalation.  A deterministic contract stops such risks early.

### Phase 5 – Webhook & Restore Contracts (Later Phases)

Subsequent phases will introduce additional contracts:

* **Webhook Contract:** record, replay and validate third‑party webhooks (Stripe, GitHub, Shopify).  Enforce idempotency, signature verification and ordering.  Good when AI agents generate or modify webhook handlers.
* **Restore Contract:** verify that backups actually restore and pass smoke tests.  Run `pg_dump`/`pg_restore` inside CI, execute smoke SQL checks (e.g. ensure `users` and `subscriptions` tables are populated), and record the result.  Only after the restore contract passes does the app get deployed.
* **Configuration Extensions:** integrate additional providers (Supabase, Neon, WorkOS entitlements) and support non‑SQL databases.

Each phase is designed to deliver value on its own and to integrate into the unified ProdVerdict engine, with consistent interfaces (CLI, GitHub Action, MCP) and shared policies.
